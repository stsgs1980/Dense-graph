Что показывает этот граф:
  • Layer 1 (submodule pointers) — 3 git-ребра, mechanism из .gitmodules. Полное покрытие.
  • Layer 3 (architectural) — каждый репо привязан к слою L0/L1/L2/L3 с layer exclusivity (G04).
  • Layer 4 (CI workflow) — verify-id-graph.yml и e2e-verifiers.yml запускаются на platform, сканируют все 4 репо через recursive checkout.
  • Layer 5 (sync status) — HEAD SHA, tags, remote-synced marker для каждого репо.
Что НЕ показывает этот граф (и не может в одном view):
  • Layer 2 (ID graph, 115 Related: edges) — слишком плотно. Показан агрегированной матрицей src×tgt prefix. Полный граф = hairball (avg degree 3.8, max 12). Для полного графа нужен отдельный интерактивный viewer с filtering по ID/edge-type — есть в verify-id-graph.js + _snapshots/id-graph-baseline.json.
  • Concurrent edits — submodule pointer bump pattern (3 atomic commits per §8.3). Это workflow, не структура — лучше в worklog.
  • Tag/release timeline — v2.5.0 → v2.6.0 (v2.6.0 через 18 мин после последнего worklog коммита). Лучше в git log.
Real coverage: ~85% структурных взаимосвязей репозиториев (с phantom node). Остальные 15% (dense ID graph) требуют отдельного viewer'а.

P-MAS_init phantom node (L4*): показывает архитектурную дыру — первый consumer стека. 2 phantom edges (submodule pointer на standards, consumer-of на platform) — это и есть работа задачи E1 (onboarding DEFERRED per user). Когда E1 будет сделан: (1) P-MAS_init получит SHA-pin в .gitmodules как 4-й submodule, (2) phantom edges станут solid, (3) node перейдёт с L4* на L4, (4) детали-панель покажет real SHA вместо «external». Граф делает architectural hole видимым — это и было целью.

Реальные архитектурные инсайты уже видны в preview:
  • STD-SKILL-001 (23 in-edges) — самый referenced ID в entire system. Каждый skill указывает на него. Это подтверждает что skill format — foundational contract.
  • STD→STD = 39 edges — большая часть графа — intra-standards. Это нормально: standards ссылаются друг на друга (STD-META-001 как root, остальные на него).
  • ZAI→STD = 25 edges — skills → standards. Основной consumer-flow: каждый ZAI skill указывает на свой governing standard.
  • RULE→RULE = 16 edges — guard rules cross-reference между собой. MONOLITH-012 (4 out) — самый connected rule (v1.3, был только что promoted).
  • TOOL→STD = 6 edges — verifiers (TOOL-VERIFY-004) → standards. 6 edges = 6 верификаторов явно ссылаются на STD'ы.
  • 0 STD→RULE/ZAI/TOOL edges — G07 enforced. Standards никогда не ссылаются «вниз» на rules/skills/tools. Layer exclusivity работает.
Dense viewer делает эти patterns видимыми без необходимости читать 60 .md файлов.
